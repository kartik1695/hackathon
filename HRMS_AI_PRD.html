System Overview

HRMS-AI is a full-stack, AI-native HR platform that replaces manual HR operations with a conversational agent. Employees interact via a chat-first interface to apply leaves, regularize attendance, request WFH, track upskilling, and query policies. Managers approve via notification CTAs without navigating dashboards.

Purpose & Problem

Problem Statement

  • HR ops (leaves, attendance, regularization, comp-off, upskilling) require multi-system navigation and manual approvals
  • Burnout and attrition risk are invisible until too late
  • Policy Q&A requires HR intervention for every single query
  • High HR toil on repetitive approval workflows

User Roles

Employee

Apply leaves, WFH, regularize attendance, track upskilling roadmap, query policies via chat

Manager

Approve team leaves / regularizations / WFH / roadmaps. View team analytics and burnout signals

HR

Policy management, org insights, penalty oversight, pre-payroll alerts

CFO

Payroll view, leave forecasts, workforce cost analytics

Architecture

Four Golden Rules

1
Django owns all writes. AI never touches DB directly.
2
Every AI action is async via RabbitMQ + Celery. Never block HTTP.
3
MCP tools are the ONLY way AI reads live data.
4
Open for extension, closed for modification (OCP) — everywhere.

User Flows

Flow 1 — Employee Applies Leave via Chat
1
Employee opens ChatPage → sends "Apply 3 days PL from May 5"
2
POST /api/ai/chat/ → ChatService.handle_turn()
3
Router detects intent leave_collection via keyword match → agents/router.py
4
leave_collector.py extracts: type=PL, from=May 5, days=3, to=May 7
5
If ambiguous → follow-up question sent (multi-turn state preserved)
6
On confirmation → mcp_tools calls apply_leave → LeaveService.apply()
7
LeaveService validates: balance ≥ days, no overlap, backdating limit, weekends excluded → creates LeaveRequest(status=PENDING)
8
post_save signal → dispatch_notification.delay() → manager WS push
9
LLM generates personalized confirmation with leave summary → returned to chat
Flow 2 — Manager Approves Leave via Notification CTA
1
Manager chat shows notification bubble: "Ravi Kumar applied 3 days PL" with Approve CTA
2
Manager clicks CTA → frontend sends chat message: "Approve leave #42"
3
Router detects intent approve_leave, extracts leave_id=42 (integer always present)
4
mcp_tools → approve_leave_request(leave_id=42) → RBAC check passes
5
LeaveService.approve() → single write_repo.update(status, approver, balance_deducted) → signal fires exactly once
6
process_leave_approval.delay() → idempotency guard (already_notified check) → WS push to employee
Flow 3 — Attendance Regularization
1
Employee: "I forgot to check out on April 28"
2
Router intent: regularize_attendance
3
create_regularization_request(date=2026-04-28, requested_check_out=18:00)
4
AttendanceService validates: within 3 working days, not weekend, not on approved leave, attempt_count < 3, no WFH conflict
5
Creates RegularizationRequest(status=PENDING) → freezes penalty for that date
6
Manager approves → AttendanceLog.status = REGULARIZED, penalty reversed if active + unlocked
Flow 4 — AI Policy Q&A (RAG)
1
Employee: "How many PLs accrue per month?"
2
Router intent: policy_query
3
rag_retrieval.py → pgvector cosine similarity search against RAGDocument table
4
Top-k chunks injected into AgentState.retrieved_docs
5
LLM builds response grounded in retrieved policy chunks — no hallucination of policy facts
Flow 5 — Skill Roadmap (AI → Manager Approval)
1
Employee uses Mentor Q&A in UpskillPage → AI generates DraftRoadmap with structured steps
2
Employee submits → SkillRoadmap(status=PENDING_APPROVAL) created
3
Manager sees pending roadmap in dashboard → approves → status → IN_PROGRESS
4
Employee marks steps complete, submits evidence URLs (submission_url)
5
Employee submits for review → PENDING_REVIEW → manager completes → COMPLETED

Functional Requirements

FR-01Multi-Provider LLM
All agent nodes use LLMProviderFactory — never direct SDK calls. Supports openai / anthropic / gemini / azure / ollama via env var LLM_PROVIDER.
Validation: Unknown provider raises ValueError. Phase 4 test: set anthropic → agents work unchanged.
core/llm/factory.py
FR-02JWT Authentication
Email + password → access + refresh token pair. Role embedded in JWT claims. Token lifetime configurable via env.
Validation: Active user, correct password, email unique constraint.
apps/employees/views.py
FR-03Leave Apply + Balance Check
Validate leave type, available balance, no overlap with existing approved leaves, backdating limit, weekends excluded before creating LeaveRequest.
Validation: balance ≥ days, no date overlap, within backdating policy, working days only.
apps/leaves/services.py:LeaveService.apply
FR-04Leave Approval (Idempotent)
Single DB update prevents duplicate signals. Idempotency guard in ProcessLeaveApprovalTask checks already_notified before pushing notification.
Validation: Requester is manager, leave is PENDING, signal fires exactly once.
tasks/leave_tasks.py:ProcessLeaveApprovalTask
FR-05Comp-Off Workflow
Employee requests CO for a worked day. Manager approval credits comp_off balance. Rejection does not credit. Double-approval blocked.
Validation: worked_date in past, not a weekend, valid employee–manager relationship.
apps/leaves/services.py:CompOffService
FR-06Daily Attendance Log Creation
Celery Beat task at 00:05 UTC creates ABSENT or ON_LEAVE log for every active employee. Status depends on approved leave coverage for the day.
Validation: Idempotent — safe to run twice. No duplicate logs per employee per date.
tasks/attendance_tasks.py:CreateDailyAttendanceLogsTask
FR-07Regularization (3-Attempt / 3-Day Window)
Employee corrects missed check-out. Maximum 3 attempts per date. Request must be within 3 working days (configurable via AttendancePolicy).
Validation: Past date, not weekend, not on approved leave, attempt_count < 3, no simultaneous WFH for same date.
apps/attendance/services.py:RegularizationService
FR-08WFH with Lead Time Enforcement
Employee applies WFH for list of dates (JSON). Min lead time enforced from AttendancePolicy.wfh_min_lead_days. Approval auto-creates/updates AttendanceLogs.
Validation: Future dates, no weekends, min lead time, no overlap with pending regularization.
apps/attendance/services.py:WFHService
FR-09PL/LOP Penalty Calculation
Daily scan (01:00 UTC) detects unresolved ABSENT logs past window. Deducts PL first then LOP (PLThenLOPStrategy). Skips dates with PENDING regularization or WFH.
Validation: Payroll lock detection on last working day of month. OCP — new strategies via PenaltyStrategyFactory.register().
apps/attendance/penalty_strategies.py
FR-10AI Intent Routing (20+ Intents)
Router classifies user query to 20+ intents. Keyword match runs before embedding calls (performance optimization). Routes to correct LangGraph flow.
Validation: Role-aware routing (manager sees approval intents, employee sees self-serve). Mid-flow policy questions handled without losing state.
agents/router.py + intent_registry.py
FR-11Multi-Turn Leave Collection
Stateful leave collection across conversation turns. Handles partial info (date missing, type ambiguous), mid-flow policy questions, and multi-type leave in one session.
Validation: State preserved in AgentState.leave_items + collection_stage. Confirmation step before submission.
agents/nodes/leave_collector.py
FR-12RAG Policy Q&A
Policy questions answered via pgvector similarity search on chunked policy documents. Retrieved chunks injected into LLM context — answers grounded in source text.
Validation: Never fabricate policy facts if tool_results absent. Source attribution in response.
agents/nodes/rag_retrieval.py
FR-13Re-Notify Manager (×3 / 60-min cooldown)
Employee re-sends reminder to manager on pending leave. Max 3 attempts. 60-min cooldown between attempts. Returns detailed status with next-available time.
Validation: renotify_count ≥ 3 → limit_reached. Within cooldown → returns next_available_in_minutes.
apps/notifications/services.py:renotify_leave
FR-14Burnout Detection
Daily scan at 02:00 UTC computes burnout score per employee from attendance patterns, overtime, leave history. Feeds into manager dashboard and chat agent.
tasks/burnout_tasks.py + agents/nodes/burnout.py
FR-15Skill Roadmap AI Generation
Mentor Q&A conversation → AI generates structured DraftRoadmap → employee submits → manager approval → step-by-step tracking with evidence URLs.
Validation: 8 skill categories auto-inferred. Status machine: PENDING_APPROVAL → IN_PROGRESS → PENDING_REVIEW → COMPLETED.
mcp/tools/upskilling_tools.py
FR-16Anonymous Feedback + Sentiment
Feedback submitted without PII. Async AI extracts: sentiment_score (−1 to 1), emotion, topics, risk_flags (burnout/attrition/morale_decline/toxic_culture). Aggregated into OrgInsightSnapshot for HR.
apps/feedback/models.py + tasks/feedback_tasks.py
FR-17WebSocket Real-Time Notifications
JWT-authenticated WS connection. Flushes ≤20 unread on connect. Incremental push on approval events. seenNotifIds ref prevents duplicate bubbles on reconnect.
apps/notifications/consumers.py + config/asgi.py
FR-18Chat Memory (Multi-Turn Context)
Redis turn cache at chat:turns:{session_id}. Tool result pinning at chat:tools:{session_id}. Entity log for pronoun resolution (focus_employee_id extracted async via Celery).
apps/ai/memory.py + apps/ai/context.py
FR-19Multi-Queue Celery Processing
Separate queues: leave, notifications, ai_heavy, analytics. Dedicated workers with different concurrency. RabbitMQ broker for durability — not Redis.
config/celery.py + docker-compose.yml
FR-20Social Feed
Employees post text + optional kudos + images. Likes with unique constraint (post + employee). Threaded comments. Paginated feed with liked_by_me annotation.
apps/feed/models.py + apps/feed/views.py

Non-Functional Requirements

Performance
  • Keyword match before embedding in router — latency reduced from 10.4s baseline
  • Pre-computed reports at 02:20 UTC — no on-demand aggregation
  • All Redis keys via CacheKeys.* — no inline key strings
  • ai_heavy queue concurrency=2 (LLM-bound — avoids OOM)
  • Separate Gunicorn (HTTP) and Daphne (WS) processes
Security
  • JWT with configurable lifetime via JWT_ACCESS_TOKEN_LIFETIME_MINUTES
  • RBAC permission class on every view (IsEmployee / IsManager / IsHR / IsCFO)
  • MCP RBAC (mcp/rbac.py) as second layer on all tool invocations
  • WS auth via JWT query param (⚠️ risk: use Sec-WebSocket-Protocol in prod)
  • Anonymous feedback: no PII stored by design
Scalability
  • Celery multi-queue + horizontal worker scaling per queue type
  • RabbitMQ broker (not Redis) — durable message delivery
  • pgvector for embedding-native search — no external vector DB
  • Redis 512MB LRU policy — cache eviction under memory pressure
  • Named bind-mount volumes — data survives docker volume prune
Observability
  • BaseHRMSTask catches all exceptions + logger.exception() — no silent failures
  • Agent graph emits per-node timing logs for performance profiling
  • [MEMORY] prefix with ANSI green for chat context debug logs
  • Celery Beat schedule visible in Django admin (DatabaseScheduler)
  • Consistent error format: {error, code: SNAKE_CASE, details: {}}

Gaps & Risks

Missing Missing Features
  • Performance reviews: models + tasks exist, no agent intents wired → dead code path
  • Payroll: PayrollSlip stores gross/net, no calculation logic → Phase 6 blocker
  • Email/Slack: handlers registered, integration completeness unknown — may silently fail outside in-app WS
  • No automated test suite — only session-tested edge cases documented in CLAUDE.md
Debt Technical Debt
  • ChatPage.tsx at ~104KB — single file, needs component extraction
  • Dashboard.tsx at ~100KB — same issue
  • agents/nodes/mcp_tools.py at 61KB — 50+ tools in one file, needs domain split
  • apps/employees/services.py at ~42KB — likely SRP violation
  • pgvector index strategy not visible — IVFFlat/HNSW config needed at scale
Risk Security & Operational Risks
  • WS auth via ?token= query param exposes JWT in server access logs — use Sec-WebSocket-Protocol header in prod
  • LLM_PROVIDER=anthropic swap test (FR-1) not verifiable without running agents end-to-end
  • Payroll lock + cross-month penalty reversal logic has no test coverage visible
  • Homebrew RabbitMQ on localhost:5672 conflicts with Docker binding — brew services stop rabbitmq required in local dev
